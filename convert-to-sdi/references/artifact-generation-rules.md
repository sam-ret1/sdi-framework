# Artifact generation rules (Phase 2)

Phase 2 generates the artifact bundle. For each artifact, this reference defines:

- **What to extract from code** (Phase 0 audit output)
- **What was confirmed by the user** (Phase 1 answers)
- **What stays as placeholder** (with confidence flag)
- **Confidence level** at file and section levels

Source flags follow `confidence-flags.md` conventions.

## Pre-check: existing artifact handling

Before generating any artifact, **check whether it already exists** in the project. If it does, the strategy chosen in Phase 1.5 (per `existing-artifact-handling.md`) governs:

- **Strategy A (preserve + header)** → don't generate from template; just add framework header to existing file. Skip the rules below for that artifact.
- **Strategy B (convert in place)** → read existing, merge with canonical template per the rules below, **show diff to user before writing**.
- **Strategy C (rename + generate)** → rename existing to `<name>.legacy.md`, then apply the rules below as if generating fresh; add note linking the legacy file at the top of the new one.
- **Strategy D (skip)** → do NOT generate; record external pointer in AGENTS.md "External artifact references" section.

The rules below apply to **fresh generation** (no existing artifact, or after rename in Strategy C). For Strategy B (convert in place), use the rules as the **target structure** and merge existing content into it.

## Generation order

Generate in this order — each builds on the previous:

```
PROJECT_STRUCTURE → ARCHITECTURE → DECISIONS (seed) → DESIGN_SYSTEM (if UI) →
PRD → ROADMAP (optional) → MEMORY init → AGENTS.md → README
```

AGENTS.md is near-last because it benefits from knowing the doc map. README is last because it indexes everything.

## Per-artifact rules

### `AGENTS.md` (root) — RICH

**File flag:** `> Source: code analysis + user-confirmed (high confidence). Generated by convert-to-sdi on YYYY-MM-DD.`

Use [`sdi-mode/adapters/claudecode-codex/AGENTS.md`](../../sdi-mode/adapters/claudecode-codex/AGENTS.md) as the base template. Customize:

| Section | Source | Treatment |
|---|---|---|
| Identity | template | unchanged |
| Project context — Type | Q1 / auto-detected | filled |
| Project context — AI modifier | Q2 / auto-detected | filled |
| Project context — Stack | Phase 0 stack detection | filled |
| Project context — Deployment target | Phase 0 detection | filled if known, else placeholder |
| Document map | known after convert-to-sdi run | filled |
| SDI discipline (8 steps) | template | unchanged |
| Project-specific conventions | Phase 0 conventions detection | filled — list each detected convention with `Source: code analysis` |
| **Production constraints** | Q3 if `mature-production` or `mvp-launched` (or auto-detected if Q3 skipped with 3+ indicators) | insert template from `production-constraints-template.md` |
| Critical context | Q5 | filled or placeholder |
| Work tracker | empty initially | empty table — Phase 3 will add the first row |
| What this is NOT | template | unchanged |
| Tone | template | unchanged |
| Document precedence | template | unchanged |
| References | template | unchanged |

If the user's project already had an `AGENTS.md` from convert-to-sdi or another source, **do not overwrite** — Phase 1.5 already determined the strategy (typically A if SDI-formatted, B if framework-aware but partial, C if user wants fresh). See `existing-artifact-handling.md` for the full handling protocol.

### `PROJECT_STRUCTURE.md` — RICH

**File flag:** `> Source: code analysis (high confidence). Generated by convert-to-sdi on YYYY-MM-DD.`

Use the type-specific template from `mvp-architect/references/project-types/{type}/project-structure-template.md` as the structure. Fill with the actual repo layout — not the template's example layout.

- **Top-level layout:** real `tree` of top-level dirs with one-line descriptions inferred from contents.
- **Per-directory sections:** for each major directory, real file tree at depth 1–2.
- **Coding conventions:** detected conventions (TypeScript strict, lint rules, test runner, commit style).
- **Convention exceptions:** if the audit found inconsistencies (mixed naming, divergent patterns), list them as **observations**, not bugs. Use language like: *"`src/legacy/` follows older patterns — appears intentional, kept as-is."*
- **Coding agent kickoff prompt template:** include the appropriate kickoff prompt at the bottom (Path A or Path B from `kickoff-prompt-template.md`).

If existing docs (`README.md`, `docs/`) already describe project structure, cross-reference rather than duplicate. Note: *"Existing structure documentation in [path] preserved; this file is the canonical layout for the framework."* If `PROJECT_STRUCTURE.md` itself already exists, follow the strategy chosen in Phase 1.5 (per `existing-artifact-handling.md`).

### `ARCHITECTURE.md` — MIXED

**If existing**, see `existing-artifact-handling.md` — default Strategy B (merge with type appendix sections), or C if outdated >1 year / wiki-style.

**File flag:** `> Source: code analysis (mixed confidence). Generated by convert-to-sdi on YYYY-MM-DD. See per-section flags.`

Use the core template from `mvp-architect/references/core-templates/architecture-template.md` + the type appendix.

| Section | Confidence | Source/treatment |
|---|---|---|
| §1 Tech Stack | high | extracted from Phase 0 |
| §2 Type-specific (multi-tenancy / data sources / ingestion / agent topology / etc.) | medium | extract what code reveals; mark sub-sections with confidence as appropriate |
| §3 Critical Flows | medium | walk through the 2–3 most prominent route/handler/loop entries; mark as "extracted from code, confirm during next work item" |
| §4 Security Considerations | low–medium | what's detectable in code (CSRF, signature verification helpers, auth gates) is high; aspirational/risk posture is placeholder |
| §5 Observability | high if present | from observability config + structured logger usage |
| §6 Integration-specific | medium | per detected major integration |
| §7 Trade-offs & Risks | low | placeholder — *"Trade-offs are typically established during initial planning; populate as discovered during work."* |

### `DESIGN_SYSTEM.md` — MEDIUM (only if UI)

**If existing**, see `existing-artifact-handling.md` — default Strategy B (extract tokens from existing). If design system lives in Figma, default Strategy D.

**File flag:** `> Source: code analysis + best-effort (medium confidence). Generated by convert-to-sdi on YYYY-MM-DD.`

Generate only if Q1/auto-detection indicates a UI-bearing type. Otherwise skip.

Extract:
- **Tokens** — from `tailwind.config.*`, CSS custom properties in `globals.css`/`tokens.css`, theme objects.
- **Typography** — fonts loaded, sizes used (scan utility-class usage statistics if Tailwind).
- **Components inventory** — list components in `src/components/` with one-line description per file.
- **Design intent** — placeholder. Aesthetic intent is rarely in code.

### `DECISIONS.md` — SEED (or ADR index)

**If existing as single `DECISIONS.md`**, see `existing-artifact-handling.md` — default Strategy A if format already matches; B otherwise.

**If ADR folder exists** (`docs/decisions/` or `docs/adr/`) — special case. Default to **ADR Option 1** (keep ADRs + generate `DECISIONS.md` as index pointing to them). User can override to **ADR Option 2** (convert ADRs to DECISIONS.md format) but rare. See `existing-artifact-handling.md` §"ADR handling".

**File flag (fresh generation):** *(at top, comment — not the file flag pattern)*
```markdown
# DECISIONS

> Append-only paper trail of non-obvious decisions. Initial entries (#1–#N) seeded by convert-to-sdi from code analysis on YYYY-MM-DD; rationale marked `?` and pending team confirmation. Future entries are written normally during work.
```

Seed 3–5 entries for **obvious patterns** detected:

```markdown
### #1 — Uses [Drizzle] as the ORM

**Context**: ORM choice for the [Postgres] layer.

**Decision**: [Drizzle].

**Rationale**: ?

**Source**: code analysis — confirm rationale with team.

**Revisit when**: ?
```

Good seed candidates:
- ORM / database client choice
- Auth provider choice
- Background-job runner (Inngest, Temporal, BullMQ, etc.)
- Test runner (vitest vs jest, etc.)
- Deployment target

Bad seed candidates (don't seed these — they require real reasoning):
- Specific library choices that don't seem load-bearing
- Naming conventions
- File-organization patterns

### `ROADMAP.md` — OPTIONAL

**If existing**, see `existing-artifact-handling.md` — default Strategy B if it has phase/quarter structure; C if pure export format (Trello/CSV); D if roadmap lives in Linear/Jira.

Generate only if Q4 = "discrete phases planned" AND the user volunteers visibility into future phases AND no existing ROADMAP requires Strategy D.

If skipped:

**File flag:** `> Source: not generated — project follows continuous-feature/maintenance cadence per Q4. Future work items are scoped one-at-a-time via Phase E of mvp-architect skill. Convert this file to a real roadmap if the project shifts to discrete-phase planning.`

If generated, follow `mvp-architect/references/core-templates/roadmap-template.md` with whatever phases the user names.

### `README.md` — INDEX (existing preserved per Phase 1.5 strategy)

If a `README.md` already exists, follow the strategy chosen in Phase 1.5 (`existing-artifact-handling.md`):

- **Strategy A (default for compatible README)** — add framework header at top, preserve all existing content:
  ```markdown
  > **Framework:** This project uses the sdi-framework workflow. See [AGENTS.md](AGENTS.md) for operating discipline.
  ```
- **Strategy B** — merge framework template (summary, stack table, doc links) with existing content; show diff before writing.
- **Strategy C** — rename existing to `README.legacy.md`, generate fresh.

If no README exists, generate a minimal one using `mvp-architect/references/core-templates/readme-template.md`, populated from convert-to-sdi outputs.

## Order of generation matters

If you generate AGENTS.md first, you don't yet know the document map (other artifacts haven't been created). The order at top of this file (PROJECT_STRUCTURE → ARCHITECTURE → DECISIONS → DESIGN_SYSTEM → PRD → ROADMAP → MEMORY → AGENTS → README) avoids that pitfall.

## Don't do

- ❌ **Don't write a polished PRD.** It's thin by design. Polish it later when you know the answers.
- ❌ **Don't seed DECISIONS for every code pattern.** 3–5 entries max. The list grows organically during work.
- ❌ **Don't infer Goals or Target users from sparse signals.** If the code has billing + auth, that doesn't tell you who the *target user* is — leave placeholder.
- ❌ **Don't generate ROADMAP unless asked.** A speculative roadmap rots immediately and confuses future readers.
- ❌ **Don't overwrite existing docs** without asking. Preserve > overwrite.
