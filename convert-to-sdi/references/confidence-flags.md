# Confidence flags

Every artifact emitted by convert-to-sdi carries a confidence flag at the top. Sections within an artifact may carry their own flag if confidence varies internally. Flags exist so the user (and future readers) immediately understand how trustworthy each piece is.

## The three levels

### High confidence — `> Source: code analysis`

The content was extracted directly from the codebase, configuration files, or unambiguous repo signals. No inference involved.

Examples:
- Stack entries (`Next.js 15 App Router` from `package.json`)
- Top-level directory layout in `PROJECT_STRUCTURE.md`
- Test runner configuration
- Existing dependencies and their versions
- Schema sketches extracted from migration files or ORM definitions

Format at top of file:
```markdown
> **Source:** code analysis (high confidence). Generated by convert-to-sdi on YYYY-MM-DD.
```

### Medium confidence — `> Source: user-confirmed`

The content was confirmed by the user in Phase 1 (or auto-detected and confirmed during the audit-report review). The user's answer is the source of truth.

Examples:
- Project type (confirmed in Q1)
- AI modifier flag (confirmed in Q2)
- Project stage (confirmed in Q3)
- Work cadence (confirmed in Q4)
- Critical context items added by the user in Q5

Format:
```markdown
> **Source:** user-confirmed (medium confidence). Confirmed during convert-to-sdi Phase 1 on YYYY-MM-DD.
```

### Low confidence — `> Source: best-effort placeholder`

The content is a placeholder. The user didn't have the answer (or skipped the question), and the code couldn't determine it. The artifact is generated thin — explicitly so the user knows to populate as they learn the project.

Examples:
- PRD §Goals (the "why" of the product)
- PRD §Target users (without a marketing site or signup analytics, often unknown)
- ARCHITECTURE §Trade-offs (rationale for past decisions, often lost)
- DECISIONS entries about non-obvious code patterns where rationale isn't documented
- ROADMAP future phases (when no roadmap exists)
- Critical context section if Q5 was skipped

Format:
```markdown
> **Source:** best-effort placeholder — populate as you learn the project (low confidence). Generated by convert-to-sdi on YYYY-MM-DD.
```

## Per-section flags

When an artifact has mixed confidence (most of it is high, but one section is low), use a section-level flag instead of (or in addition to) a file-level flag.

Example in `ARCHITECTURE.md`:

```markdown
## 1. Tech Stack

> Source: code analysis (high confidence)

[stack table extracted from manifest + config files]

## 4. Critical Flows

> Source: code analysis (medium confidence) — extracted from route handlers and exported flow definitions; some inferred. Confirm during first work item.

[flow walkthroughs]

## 7. Known Trade-offs & Risks

> Source: best-effort placeholder — populate as you learn the project. Trade-offs are typically established during planning; this artifact was generated post-hoc, so this section is intentionally empty until populated.

(empty until user adds entries)
```

## How flags appear in DECISIONS.md

DECISIONS entries seeded by convert-to-sdi carry the flag inline:

```markdown
### #1 — Uses Drizzle as the ORM

**Context**: ORM choice for the Postgres layer.

**Decision**: Drizzle.

**Rationale**: ?

**Source**: code analysis — confirm rationale with team.
```

The `?` in Rationale is intentional. Future user (or current user, when they remember) replaces it. The `Source` line marks this as a seed entry, not a decision actively reasoned through.

## How to update flags later

The user (or the coding agent during work items) updates flags as confidence improves:

- A `low-confidence` placeholder gets filled in → upgrade to `user-confirmed` (medium)
- A `medium-confidence` user assertion gets validated by code reality → upgrade to `code analysis` (high)
- The reverse: if a `high-confidence` extraction turns out wrong → downgrade and mark for revision

This isn't bureaucracy — it's how the framework remains honest about what it knows.

## When NOT to use flags

- **Don't flag DECISIONS entries written by the user during normal work** (post-convert-to-sdi). Those are normal entries, not seeds. The flag exists specifically to mark `convert-to-sdi` outputs.
- **Don't flag MEMORY entries.** Memory is always datable and inherently best-effort recollection — flags would be noise.
- **Don't flag PROJECT_STRUCTURE drift notes.** Those are observations, not assertions.

## Smoke test for flags

After convert-to-sdi runs, every generated artifact should have a flag at the top (or section flags where appropriate). If a file is unflagged, you missed it. Sweep:

- README.md — flag at top
- AGENTS.md — flag at top, plus per-section flags for `Stack`, `Project-specific conventions`, `Production constraints`
- PROJECT_STRUCTURE.md — flag at top
- ARCHITECTURE.md — flag at top, plus per-section flags
- DECISIONS.md — entries individually marked with `Source: code analysis — confirm rationale with team`
- docs/MEMORY.md — no flag (memory is always datable)
- docs/memory/YYYY-MM-DD.md — no flag (memory is always datable)
- PRD.md — flag at top, plus low-confidence flags on Goals/Users sections
- ROADMAP.md — flag at top (low confidence if generated empty)
- DESIGN_SYSTEM.md (if present) — flag at top
- IMPLEMENTATION_PLAN_*.md (Phase 3 output) — flag in §0 Pre-requisites noting that the plan was generated against current production state (not a Phase N-1 delivery)
